# CLI Login System

A command-line authentication system written in Go: registration, login,
optional TOTP-based 2FA (Google Authenticator compatible), account lockout,
and session management with a configurable timeout. Runs in Docker with
Postgres for persistence, and all SQL is generated via **sqlc** for
type-safe query code.

## Stack

- **Language:** Go 1.22
- **Database:** PostgreSQL, accessed via `jackc/pgx/v5` (pure-Go driver, no cgo)
- **Query layer:** [sqlc](https://sqlc.dev) — hand-written SQL in `db/queries/`
  and `db/schema/`, compiled into type-safe Go in `internal/dbgen/`
- **Password hashing:** bcrypt (`golang.org/x/crypto/bcrypt`)
- **2FA:** TOTP, Google Authenticator compatible (`pquerna/otp`)
- **CLI:** interactive REPL with history and tab-completion (`chzyer/readline`)

## Project layout

```
cmd/cli/main.go             entrypoint: config + wiring
db/schema/001_init.sql      Postgres schema — source of truth for sqlc + migrations
db/queries/*.sql            hand-written SQL, annotated for sqlc
sqlc.yaml                   sqlc configuration
internal/dbgen/             generated by `sqlc generate` — do not edit by hand
internal/models/            plain domain structs (User, Session) used outside the DB layer
internal/db/                Postgres connection pool + applies db/schema on startup
internal/auth/              registration, login, lockout, TOTP (business logic)
internal/session/           session creation/validation/expiry
internal/cli/               the interactive REPL and command dispatch
```

The schema is applied automatically on startup (idempotent `CREATE TABLE IF
NOT EXISTS`) — no separate migration step required to run the app.

## Regenerating query code

If you change `db/schema/*.sql` or `db/queries/*.sql`, regenerate
`internal/dbgen/`:

```bash
# install sqlc if you don't have it: https://docs.sqlc.dev/en/latest/overview/install.html
sqlc generate
```

## Running with Docker (recommended)

`docker compose up` streams logs but does **not** forward your terminal's
input to the container — for an interactive program like this one you want
`docker compose run` instead, which attaches stdin/stdout/tty properly:

```bash
docker compose run --build --rm cli-login
```

`--build` makes sure the image reflects any local changes; `--rm` removes
the container on exit. This still starts Postgres first and waits for its
healthcheck, per the `depends_on` in `docker-compose.yml`.

If Postgres is already running (e.g. from a previous `up`), you can skip
straight to:

```bash
docker compose run --rm cli-login
```

Postgres data lives in the `postgres-data` named volume, and CLI history in
`login-data` — both survive container restarts and re-runs (only
`docker compose down -v` removes them).

## Running locally (without Docker)

Requires Go 1.22+ and a reachable Postgres instance.

```bash
export DATABASE_URL="postgres://user:pass@localhost:5432/dbname?sslmode=disable"
go build -o cli-login ./cmd/cli
./cli-login
```

## Usage

On start you'll see a `login>` prompt.

**Before logging in:**

| Command    | Description                              |
|------------|-------------------------------------------|
| `register` | Create a new account (prompts for username/password) |
| `login`    | Log in (prompts for username/password, then a TOTP code if 2FA is enabled) |
| `help`     | List available commands                   |
| `exit`     | Quit                                       |

**After logging in** (prompt becomes `<username>>`):

| Command       | Description                                   |
|---------------|------------------------------------------------|
| `whoami`      | Show account details (see below)               |
| `enable-2fa`  | Generate a TOTP secret, display it + a QR-able `otpauth://` URL, and confirm with a code |
| `disable-2fa` | Turn off 2FA                                   |
| `logout`      | End the session                                |
| `help`        | List available commands                        |
| `exit`        | Log out and quit                                |

User details (shown automatically right after login, and via `whoami`):
username, registration date, 2FA status, session expiration time, and last
login time (or "first login").

### Enabling 2FA

`enable-2fa` prints a secret and an `otpauth://` URL. Add it to Google
Authenticator (or any TOTP app) either by scanning a QR code generated from
that URL (e.g. via any QR-code generator, since this is a terminal app) or by
entering the secret manually. Then enter the current 6-digit code to confirm
— 2FA isn't active until that confirmation succeeds.

## Security behavior

- Passwords are hashed with bcrypt (cost configurable, default 12) — never stored or logged in plaintext.
- Passwords must be 8+ characters with at least one letter and one digit.
- Usernames must be 3–32 alphanumeric/underscore characters.
- After `MAX_LOGIN_ATTEMPTS` consecutive failures (default 5), the account is
  locked for `LOCKOUT_MINUTES` (default 15). A wrong TOTP code also counts as
  a failed attempt.
- Login failure messages don't reveal whether a username exists.
- Sessions are server-side tokens with an expiry (`SESSION_TIMEOUT_MINUTES`,
  default 15); the CLI checks expiry before each command and forces re-login
  once a session lapses.

## Configuration

All configuration is via environment variables (see `docker-compose.yml` for
the defaults wired up):

| Variable                    | Default                                                            | Description                          |
|-----------------------------|---------------------------------------------------------------------|---------------------------------------|
| `DATABASE_URL`               | `postgres://postgres:postgres@localhost:5432/postgres?sslmode=disable` | Postgres connection string           |
| `HISTORY_FILE`               | `/data/.cli_login_history`                                          | Readline history file                 |
| `MAX_LOGIN_ATTEMPTS`        | `5`                                                                   | Failures before lockout               |
| `LOCKOUT_MINUTES`           | `15`                                                                  | Lockout duration                      |
| `SESSION_TIMEOUT_MINUTES`   | `15`                                                                  | Session validity window               |
| `BCRYPT_COST`               | `12`                                                                  | bcrypt work factor                    |
| `TOTP_ISSUER`               | `CLI Login System`                                                   | Issuer name shown in authenticator apps |

## Tests

Tests need a running Postgres, pointed to by `TEST_DATABASE_URL`:

```bash
export TEST_DATABASE_URL="postgres://postgres:postgres@localhost:5432/clitest?sslmode=disable"
go test ./...
```

They're skipped automatically (not failed) if `TEST_DATABASE_URL` is unset.
Covers registration/validation rules, login, account lockout and its expiry,
the full TOTP enable → confirm → require-on-login → disable flow, and
session creation/validation/expiry/destruction — all run against a real
Postgres, truncating tables between tests for isolation.
